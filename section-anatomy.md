# Brand source

Brand isn't a single config. It's whichever **one** of the following the
user already has, in this order of preference:

1. **An existing brand skill** (e.g. `acme-brand`, `company-design-system`)
2. **A brand reference file** (`brand.json`, `brand.md`, etc.) in the
   working directory or uploads
3. **A scaffolded `brand.json`** generated by this skill, only if 1 and 2
   are unavailable

The point: **do not invent brand values**. Check first. Generic blue
fallback colours sneaked into a section script is the kind of bug that
makes the report look unprofessional and that nobody catches until it's
in front of the CEO.

## Step 1 — Look for an existing brand skill

Use the available-skills mechanism (in claude.ai, this is the
`<available_skills>` list; via API, `tool_search` for "brand"). Look for
skills whose name ends `-brand` or contains "design system" or "brand
system" in the description.

If you find one, `view` its SKILL.md and extract:

- **Primary colour** — header backgrounds, page headers (typically dark)
- **Secondary colour** — buttons, accents, H3 headings
- **Accent colour** — section labels, stat values, callout borders
- **Alert colour** — adverse variances, urgent flags
- **Warning colour** — caution states
- **Text colours** — body text, muted text, subtle text
- **Border/background colours** — borders, page background, card background
- **Font family + fallback** — single primary font, system fallback
- **Logo path** — absolute or relative path to a logo file
- **Style rules** — em-dash policy, case rules for section labels, etc.

Record these values either:

- Inline in each renderer's brand-constants block (the simpler option),
  or
- In a derived `brand.json` cached in the working directory, with each
  renderer reading from it.

The reference implementation this skill is extracted from uses inline
constants — every renderer
has `NAVY = "#004673"` etc. at the top. This works because the brand is
stable. If the brand might change, prefer the JSON-cache approach.

## Step 2 — Look for a brand reference file

If no skill, scan for:

- `brand.json`, `brand.md`, `brand-guidelines.md`
- A `brand/` or `branding/` directory
- Files in `/mnt/user-data/uploads/` named like the above

If JSON, parse directly. If markdown, extract the colour table and font
spec, then derive a `brand.json` for the renderers to reference.

## Step 3 — Ask the user

If neither exists:

> "I don't see a brand skill or brand reference in your workspace. Do
> you have one (a colour palette, font, logo) I should use? If you can
> share the colours and font, I'll write them up. Otherwise I can
> scaffold a starter `brand.json` with neutral defaults — you can edit
> the colours and we'll proceed."

Wait for the response. Do not auto-generate without asking.

## Step 4 — Scaffold (only if asked to)

Copy `assets/brand.example.json` to the working directory as
`brand.json`. Walk the user through the fields:

- Primary, secondary, accent, alert colours (hex)
- Text colours (body, muted, subtle)
- Background and border colours
- Font family (must be web-loadable — Google Fonts, system, or
  bundled `.ttf`)
- Logo path (if they have one — leave null otherwise)

For colours, sensible neutral defaults:

| Token | Default |
|---|---|
| primary | `#1F3A5F` (deep navy) |
| secondary | `#3D6FAB` (mid blue) |
| accent | `#4A9DDB` (sky blue) |
| alert | `#C0392B` (red) |
| warning | `#E67E22` (amber) |
| text | `#535B62` (charcoal) |
| text_muted | `#727C87` (mid grey) |
| border | `#DFE3E5` (pale grey) |
| bg_page | `#F4F6F8` (near white) |
| bg_card | `#FFFFFF` |

These are deliberately bland — they'll produce a clean-looking report
that nobody loves but nobody objects to. A real brand will look better.

## Where brand values flow

| Consumer | How it picks up brand |
|---|---|
| Section renderers | Module-level constants at the top of `renderer.py` |
| `assemble_report.py` | Module-level constants near the top (cover page motifs, global footer, page header) |
| `chart_helper.py` | Module-level constants (palette for bar/line charts) |
| Section CSS | Hex values inlined in scoped `<style>` blocks |
| Assembled HTML CSS | Hex values inlined in the `BRAND_CSS` constant |

There is no runtime CSS variable injection. The reason: WeasyPrint has
historically been less reliable with CSS custom properties than with
direct hex values, and direct hex makes every fragment renderable in
isolation in a browser without an outer document defining the variables.

If you want a single point of truth, derive a `brand.py` module that
all renderers import:

```python
# brand.py — synced from brand source
NAVY     = "#004673"
BLUE     = "#1C72B5"
SKY      = "#37A6DE"
# etc.
```

Then renderers do `from brand import NAVY, BLUE, SKY` instead of
hard-coding. This is optional — many sections just hard-code.

## Style rules beyond colour

| Rule | Where it applies |
|---|---|
| All section labels uppercase | Every `.kl`, `.sl`, section title |
| No em dashes (en dashes or colons instead) | **Auto-commentary only** — user commentary is verbatim |
| Single font family throughout | Every renderer and assembler |
| Currency formatting | `$X,XXX`, `$X.Xk`, `$X.XM` patterns |

The em-dash rule is the most often-broken. It applies to your own prose
in auto-commentary. It **does not apply** to user commentary, which is
rendered exactly as given. See `commentary-verbatim.md`.

## Logo handling

The assembler embeds the logo as a base64 inline image so the HTML is
self-contained. It tries to find the logo in this order:

1. Path in the brand skill's assets directory
2. Path in `brand.json -> company.logo_path`
3. Falls back to a text "Company Name" placeholder if no logo found

If no logo is available, the report still renders — the header logo box
becomes a small text-only badge. Don't block the run for a missing logo.
